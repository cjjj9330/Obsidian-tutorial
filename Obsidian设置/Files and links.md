### 1. Confirm file deletion

确认文件删除。删除文件前进行提示确认。启用此选项后，当你尝试删除一个文件时，Obsidian会弹出确认对话框要求你确认删除操作，以防止意外删除重要文件

### 2. Deleted files

已删除文件，设置文件删除后的处理方式。
- Move to system trash（移至系统回收站）：删除的文件将被移动到系统回收站
- Move to Obsidian trash (.trash folder)（移至Obsidian回收站）：删除的文件将被移动到笔记库中的.trash文件夹
- Permanently delete（永久删除）：删除的文件将被直接永久删除，无法恢复

### 3. Automatically update internal links

重命名文件后自动更新链接。启用此选项后，当你重命名一个文件时，Obsidian会自动更新所有指向该文件的内部链接，以保持链接的有效性。如果关闭此选项，在重命名文件时会提示你是否要更新链接

### 4. Default location for new notes

设置新建笔记的默认保存位置。
- Same folder as current file（与当前文件相同文件夹）：新笔记将保存在当前打开笔记所在的文件夹中
- In the folder specified below（在下方指定的文件夹中）：新笔记将保存在你指定的特定文件夹中
- Root folder（根文件夹）：新笔记将保存在笔记库的根目录

## 5. New link format

设置自动生成内部链接时使用的格式。比如在你从文件浏览器拖拽文件到笔记编辑器中或使用 `[[` 自动完成来插入链接时：
- Shortest path when possible（尽可能使用最短路径）：使用到目标笔记的最短可能路径
- Relative path to file（相对于文件的路径）：使用相对于当前笔记的路径
- Absolute path in vault（库中的绝对路径）：使用从库根目录开始的完整路径

这里大家可能不懂，所以我又帮大家问了一次AI，这三种模式的具体区别，如下：

> [!info]+ 新链接的三种呈现选项
> 
> 让我为你解释一下这三个选项的区别：
> 
> 1.  **Shortest path when possible (尽可能使用最短路径)**：
>     * **说明**：选择此选项后，Obsidian 会尝试使用最短的路径来引用目标笔记。如果笔记名称在你的整个库中是唯一的，那么链接通常就只是笔记的名称（例如 `[[笔记名称]]`）。如果存在同名笔记，Obsidian 会添加足够多的父文件夹名称来区分它们，以确保链接指向正确的笔记，同时保持路径尽可能短。
>     * **例如**：如果你有一个名为 "会议记录" 的笔记，并且它是库中唯一一个叫这个名字的笔记，那么链接就是 `[[会议记录]]`。如果你在不同的文件夹下有两个都叫做 "项目A" 的笔记，比如 "项目/项目A" 和 "存档/项目A"，那么链接可能会是 `[[项目/项目A]]` 和 `[[存档/项目A]]`。
>     * **优点**：链接通常更简洁易读。
>     * **缺点**：如果将来移动文件或更改文件夹结构，且文件名不够独特，链接可能需要 Obsidian 更新，或在某些情况下可能导致混淆（尽管 Obsidian 通常能很好地处理这些）。
> 
> 2.  **Relative path to file (相对于文件的路径)**：
>     * **说明**：选择此选项后，链接会根据当前笔记相对于目标笔记的位置来生成。它会使用像 `../` （返回上一级目录）和 `./` （当前目录）这样的相对路径指示符。
>     * **例如**：如果你在 "项目/报告/" 文件夹下的笔记A中链接到同一文件夹下的笔记B，链接可能是 `[[./笔记B]]` 或 `[[笔记B]]`。如果你链接到上一级 "项目/" 文件夹中的笔记C，链接可能是 `[[../笔记C]]`。
>     * **优点**：如果将包含相互链接的一组文件（例如一个项目文件夹）移动到库中的其他位置，或者移动到另一个库，只要它们的相对结构保持不变，链接通常仍然有效。
>     * **缺点**：如果单独移动某个文件，而不移动它所链接的文件，或者更改了文件夹结构，链接更容易断开。链接本身可能看起来不那么直观，因为它包含了路径信息。
> 
> 3.  **Absolute path in vault (库中的绝对路径)**：
>     * **说明**：选择此选项后，链接将始终包含从你的 Obsidian 库（Vault）的根目录开始到目标笔记的完整路径。
>     * **例如**：无论你从哪个笔记创建链接，如果目标笔记位于 "文件夹A/子文件夹B/笔记X.md"，那么链接始终是 `[[文件夹A/子文件夹B/笔记X]]`。
>     * **优点**：链接的指向非常明确，无论你如何移动包含该链接的源文件，只要目标文件仍在库中的那个绝对位置，链接就有效。
>     * **缺点**：链接通常更长，可读性可能稍差。如果目标文件本身被移动到库中的其他位置，链接就需要更新。
> 
> **总结一下如何选择（这里是我写的）：**
> 
> 追求简洁和易读性，选择尽可能使用最短路径；如果要将文件夹移动到另一个库，使用相对路径；只在库中移动文件，且想确保路径准确性，选择绝对路径

### 6. Use Wikilinks

自动为链接和图片生成Wiki链接，而不是Markdown链接和图片。启用此选项后，Obsidian将使用`[[页面名称]]`格式创建链接，禁用则使用标准Markdown链接格式`[页面名称](路径)`。

在具体解释这个设置之前，我们先来看看wikilinks是什么：

> [!info]- Wikilinks是什么 ^wiki
> 好的，我们来详细解释一下 **Wikilinks** 是什么。
> 
> 简单来说，**Wikilinks**（也常被称为“维基链接”）是一种**非常简洁和直观的创建内部链接的语法**。它们最初因在维基百科（Wikipedia）这类协作式知识库网站中的使用而广为人知，现在也被许多个人知识管理工具（比如 Obsidian）所采用。
> 
> **核心特点和语法：**
> 
> 1.  **简洁的语法**：最常见的 Wikilink 语法就是用双层方括号 `[[ ]]` 将你想要链接到的页面或笔记的名称括起来。
>     * **例如**：如果你有一个笔记叫做“会议记录”，那么一个指向它的 Wikilink 就是 `[[会议记录]]`。
> 
> 2.  **自动链接**：当你输入这种格式后，支持 Wikilinks 的软件（如 Obsidian）通常会自动将其识别为一个内部链接。点击这个链接，软件就会带你跳转到名为“会议记录”的那个笔记。
> 
> 3.  **创建新页面**：如果被 `[[ ]]` 括起来的页面尚不存在，很多支持 Wikilinks 的系统（包括 Obsidian）允许你通过点击这个“空链接”来快速创建这个新页面。这使得建立新的相关主题变得非常流畅。
> 
> 4.  **别名/自定义显示文本 (可选)**：有时你可能希望链接显示的文本与实际的笔记文件名不同。Wikilinks 通常支持通过竖线 `|` 来实现这一点。
>     * **例如**：`[[实际笔记文件名|你想显示的文本]]`。如果你有一个笔记叫“2025年第一季度销售报告”，但你只想在链接中显示为“Q1销售报告”，你可以写成 `[[2025年第一季度销售报告|Q1销售报告]]`。
> 
> **为什么使用 Wikilinks？**
> 
> * **快速便捷**：输入 `[[笔记名称]]` 比输入完整的 Markdown 链接 `[显示文本](文件路径/笔记名称.md)` 要快得多，也更简单。
> * **促进连接思维**：由于创建链接非常容易，它鼓励用户在不同的想法和笔记之间建立更多的联系，有助于构建网状的知识结构。
> * **易于阅读和维护**：Wikilinks 通常直接显示笔记的名称（或你选择的别名），使得链接的意图更加清晰。
> 
> **与标准 Markdown 链接的区别：**
> 
> * **Wikilink**: `[[我的笔记]]`
> * **Markdown 链接**: `[我的笔记](路径/我的笔记.md)`
> 
> 可以看到，Wikilinks 省略了显式的路径声明和 Markdown 链接中方括号与圆括号的区分。在 Obsidian 这样的环境中，软件会在后台处理路径解析的问题。
> 
> 总而言之，Wikilinks 是一种以用户友好和高效为设计目标的内部链接机制，特别适合用于快速构建和导航个人或协作的知识库。在 Obsidian 中，它是核心功能之一，因为它极大地简化了在笔记之间穿梭和建立关系的过程。

现在就能更容易地理解这个设置了：

这个设置决定了当你插入指向其他笔记或图片的链接时，Obsidian 默认会使用哪种链接的语法格式。

这里有两种主要的链接格式：

1. **Wikilinks 格式**：
    
    - **语法**：`[[页面名称]]` 或 `[[文件名]]`
    - **说明**：这是一种非常简洁的链接方式，常见于许多知识管理软件（比如 Wikipedia）。你只需要用双中括号把笔记的名称（或者你想要显示的文本，格式为 `[[笔记文件名|显示文本]]`）括起来。
    - **例如**：如果你想链接到一个名为 "我的笔记" 的文件，Wikilink 就是 `[[我的笔记]]`。
2. **Markdown 链接格式**：
    
    - **语法**：`[显示的文本](文件路径/文件名.md)`
    - **说明**：这是 Markdown 语言标准的链接方式。方括号 `[]` 中是链接显示的文本，圆括号 `()` 中是实际的文件路径。
    - **例如**：链接到 "我的笔记.md" 这个文件，Markdown 链接可能会是 `[我的笔记](我的笔记.md)` 或者，如果它在某个文件夹里，就是 `[我的笔记](文件夹/我的笔记.md)`。

**这个设置的作用：**

- **如果启用“Use Wikilinks”（打开开关）**：
    
    - 当你通过拖拽文件、使用 `[[` 自动建议等方式创建内部链接或插入图片时，Obsidian 会自动使用 `[[文件名]]` 这种 Wikilink 格式。
    - 这对快速创建链接非常方便，也是 Obsidian 的特色之一，因为它使得笔记之间的连接更加流畅和自然。
- **如果禁用“Use Wikilinks”（关闭开关）**：
    
    - Obsidian 在自动生成链接或插入图片时，会使用标准的 Markdown 链接格式，即 `[显示文本](路径)`。
    - 这种格式的好处是它在其他不支持 Wikilinks 的 Markdown 编辑器或平台中具有更好的兼容性。如果你计划在 Obsidian 之外使用你的笔记，或者希望保持最广泛的 Markdown 兼容性，你可能会选择禁用 Wikilinks。

**简单来说：**

- **启用 Wikilinks**：链接更简洁，输入更快，更符合 Obsidian 的“连接思维”理念。
- **禁用 Wikilinks (即使用 Markdown 链接)**：链接格式更传统，与其他 Markdown 工具的兼容性最好。

Obsidian 即使在你选择了其中一种格式作为默认设置后，通常也能识别和处理另一种格式的链接。这个设置主要是关于 Obsidian _自动生成_ 链接时采用哪种风格。

大多数 Obsidian 用户倾向于启用 Wikilinks，因为它们更简洁，并且 Obsidian 对这种格式有很好的支持（比如自动更新链接、显示反向链接等）。但选择哪种取决于你的个人偏好和工作流程。

### 7. Detect all file extensions：

显示任何扩展名的文件，即使Obsidian无法原生打开它们。启用后，你可以在文件浏览器中看到所有文件，点击一般会使用对应的外部程序打开

### 8. Default location for new attachments：

设置新上传附件的默认保存位置。
- Same folder as current file（与当前文件相同文件夹）：附件将与引用它的笔记保存在同一文件夹
- In the folder specified below（在下方指定的文件夹中）：附件将保存在你指定的特定文件夹中，选择此项，下方会出现“Attachment folder path（附件文件夹路径）”，可以在这里指定文件夹存放附件
- In subfolder under current folder（当前文件夹的子文件夹中）：附件将保存在当前笔记所在文件夹的一个子文件夹中。选择此项，Obsidian会在你上传附件的时候，自动在当前文件夹中创建一个子文件夹用于保存附件。如果你想指定这个子文件的名称，可以在选择这个选项后，在下方的“Subfolder name（子文件夹名称）"中指定

### 9. Excluded files： 

这个主要是说在搜索的时候排除的文件。被排除的文件不管你在哪里搜索，一般都不会在出现在搜索结果中。可以通过右边的“Manage"按钮管理被排除的文件

### 10. Override config folder：

覆盖配置文件夹。使用与默认配置文件夹不同的文件夹。该文件夹必须以点开头，图片显示设置为".obsidian"。这个设置允许你指定一个自定义的文件夹来存储Obsidian的配置文件，而不是使用默认的.obsidian文件夹。一般来说不用动这个
